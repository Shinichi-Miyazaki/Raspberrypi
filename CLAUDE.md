# Claude Code ガイドライン

このファイルはClaude Codeがこのリポジトリを編集する際の方針を定めたものです。

---

## コーディング方針

### 可読性を最優先にする
- 短く書けても、読んでわかりにくいコードは避ける
- 変数名・関数名は何をするものか一目でわかる名前にする
  - 悪い例: `d`, `tmp`, `x`
  - 良い例: `dht_device`, `temperature`, `elapsed_minutes`
- 1つの関数には1つの役割だけを持たせる
- 複雑な処理には日本語コメントで「何をしているか」を説明する

### コメントの書き方
- 「何をしているか」が一目でわからない処理にはコメントをつける
- 「なぜそうしているか」（設計上の理由や注意点）は積極的にコメントに残す
  ```python
  # use_pulseio=False はadafruit_circuitpython_dhtと相性が悪いため省略
  dht_device = adafruit_dht.DHT22(DHT_PIN)
  ```
- コメントはすべて日本語で書く
- 自明な処理にはコメント不要（`i += 1  # 1を足す` のような冗長なコメントは書かない）

### エラー処理
- エラーが起きたときに何が原因かわかるメッセージをログに残す
- 握りつぶし（`except: pass`）はなるべく避け、最低でもログを出す

## ファイル・ドキュメント方針

### 新規ファイルは必要最小限に
- Markdownファイル（`.md`）は原則として新規作成しない
- ドキュメントは既存の `README.md` を更新するかたちで対応する
- 設定例・使い方・トラブルシューティングはすべて `README.md` にまとめる

### コードファイルについて
- 既存のファイルを編集することを優先する
- 新しいファイルを作るのは、明確に役割が異なる機能を切り出す場合に限る

---

## 説明・応答の方針

- 説明はすべて日本語で行う
- 専門用語には補足を加え、初心者にもわかるように書く
  - 悪い例: 「`axhline` で閾値ラインを追加しました」
  - 良い例: 「グラフに閾値ライン（上限・下限を示す横線）を追加しました。`axhline` はmatplotlibで水平線を引く関数です」
- 変更内容は「何を・なぜ変えたか」をセットで説明する

---

## このリポジトリ固有の注意点

- `TEST_MODE = False` がデフォルト。テスト実行時は `--test-mode` フラグを使う
- Slack のトークンとチャンネル名は環境変数（`SLACK_TOKEN`, `SLACK_CHANNEL`）で渡す。コードに直書きしない
- DHT22センサーは `use_pulseio=False` を指定しない（adafruit_circuitpython_dhtと相性が悪いため）
- GPIO の初期化・クリーンアップは必ず行う（初期化漏れで「unable to set line to input」エラーが発生する）
